## Boris Tane 프로 팁 10선

### 1. 코드 작성 전 반드시 계획 검토

```
❌ "대시보드를 만들어줘"
✅ "먼저 코드베이스를 분석하고 research.md를 작성해줘"
```

복잡한 기능일수록 리서치 → 계획 → 검토 → 구현 순서를 지키세요. 코드 작성은 전체 과정의 마지막 25%에 불과합니다.

### 2. 결과를 마크다운 파일로 작성시키기

```
❌ "분석 결과를 알려줘" (대화에 섞여 사라짐)
✅ "분석 결과를 research.md에 작성해줘" (파일로 남아 검토 가능)
```

구두 요약은 지나가면 찾을 수 없습니다. 파일로 작성하면 라인 단위로 검토하고, 수정을 요청하고, 다음 단계에서 참조할 수 있습니다.

### 3. 인라인 주석으로 계획 수정

```
❌ "계획이 좀 별로야. 다시 해줘."
✅ plan.md에 직접 "CHANGE: 이 부분은 LEFT JOIN 대신 서브쿼리를 사용해줘" 추가
```

구체적인 위치에 구체적인 피드백을 남기면, Claude가 정확히 어디를 어떻게 수정해야 하는지 즉시 파악합니다.

### 4. 기존 구현을 참조시키기

```
❌ "이 프로젝트 스타일에 맞게 해줘"
✅ "lib/queries/todos.ts의 getTodos 함수와 동일한 패턴을 따라줘"
```

구체적인 파일 경로와 함수 이름을 지목하면, Claude가 해당 코드를 읽고 동일한 패턴으로 새 코드를 생성합니다.

### 5. 수정 피드백은 간결하게

```
❌ "components/stats/category-chart.tsx 파일의 바 차트에서 className에 있는
    배경색을 좀 더 진한 색으로 변경해 주세요. 현재 bg-blue-200을 사용하고
    있는데 bg-blue-500 정도가 좋을 것 같습니다."

✅ "바 차트 색상이 너무 연해. 더 진하게."
```

Claude는 전체 컨텍스트를 가지고 있으므로, 파일 경로와 클래스 이름까지 지정할 필요가 없습니다.

### 6. 2번 실패하면 revert 후 재시도

```
1회차: "바 차트가 깨져. 수정해줘." → 수정했지만 여전히 깨짐
2회차: "아직 깨져. 다시 수정해줘." → 더 복잡해짐
3회차 ❌: "또 깨져. 한 번 더..." → 패치 위의 패치

3회차 ✅: /rewind 후 "바 차트를 다른 접근으로 구현해줘. Tailwind flex 대신
         grid로 해줘."
```

작은 패치를 쌓아가면 코드가 복잡해지고 Claude의 컨텍스트도 혼란스러워집니다. 2번 실패 후에는 깨끗한 상태에서 다시 시작하는 것이 효율적입니다.

### 7. Cherry-pick — Claude의 제안 중 좋은 것만 선택

Claude가 여러 파일을 수정했지만, 일부만 마음에 들 수 있습니다.

```
"category-chart.tsx의 변경은 좋아. 하지만 priority-chart.tsx의 변경은 원래대로 되돌려줘."
```

Claude의 제안을 전부 수용하거나 전부 거부할 필요 없이, 좋은 부분만 선택적으로 유지할 수 있습니다.

### 8. 하나의 긴 세션으로 진행

```
❌ 리서치 세션 → /clear → 계획 세션 → /clear → 구현 세션
✅ 리서치 → 계획 → 주석 → 구현을 하나의 세션에서 진행
```

Boris Tane 워크플로는 하나의 긴 세션에서 진행하는 것이 효율적입니다. Claude가 리서치와 계획의 맥락을 유지한 채 구현할 수 있기 때문입니다. research.md와 plan.md가 파일로 남아있으므로 `/clear` 후에도 참조할 수 있지만, 대화의 맥락은 사라집니다.

### 9. /clear 대신 /compact 활용 (긴 세션에서)

```
✅ /compact
```

긴 세션에서 컨텍스트가 커지면 `/compact`로 이전 대화를 요약합니다. `/clear`처럼 맥락을 완전히 잃지 않으면서 컨텍스트를 절약합니다.

{% hint style="info" %}
**`/compact`의 동작**: 이전 대화 내용을 AI가 요약하여 압축합니다. 핵심 결정사항과 현재 상태는 유지하면서 불필요한 중간 과정을 제거합니다.
{% endhint %}

### 10. 검증 기준을 먼저 정의

→ *이 프롬프트는 구현 요청 시 검증 기준을 함께 제시하여, Claude가 처음부터 품질 기준을 고려하며 코드를 작성하도록 유도하기 위한 것입니다:*

```
대시보드를 만들어줘. 완성 후 다음을 검증해줘:
1. pnpm build 통과
2. 통계 수치가 실제 데이터와 일치
3. 빈 데이터 처리
4. 반응형 레이아웃
```

검증 기준을 미리 정의하면 Claude가 처음부터 해당 기준을 고려하며 구현합니다.
